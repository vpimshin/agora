# 17 — Testing Strategy

> Test pyramid + LLM eval pyramid. Fast at the bottom, scarce and expensive at the top.

## 1. Pyramid

```
                  ▲
                  │   E2E (Playwright) — 30+ critical journeys
                  │
                  │   Component (Storybook + axe + interactions) — 100+
                  │
                  │   Integration (Vitest + supabase-js + pgtap) — 200+
                  │
                  │   Unit (Vitest, fast-check property tests) — 1000+
                  ▼
```

A parallel **LLM eval pyramid** runs alongside (see [16](16_OBSERVABILITY_AND_EVALS.md)):

```
   ▲ red-team & adversarial   (weekly)
   │ workflow / journey replay (per release)
   │ behavioural agent evals   (per PR)
   ▼ unit prompt evals         (per PR)
```

---

## 2. Unit tests (Vitest)

- Pure functions: state machine guards, FSRS scheduling, BKT updates, token math, redaction utilities.
- **Property-based tests** with `fast-check`:
  - FSRS: monotonicity (Easy ≥ Good ≥ Hard ≥ Again interval).
  - Token ledger: sum of deltas equals materialised balance for any sequence.
  - State machine: from any reachable state, exactly the documented transitions are valid.
- Coverage gate: ≥ 80 % statements, ≥ 70 % branches.

---

## 3. Integration tests

- Postgres-in-Docker via `testcontainers`.
- Real schema + RLS policies applied.
- Supabase JS client used as the system under test.
- **`pgtap`** suite for RLS:
  - tenant A cannot read tenant B's `messages`.
  - mentor cannot read raw transcripts.
  - SECURITY DEFINER token RPC enforces caller-tribe match.
- Edge functions tested via Deno test runner.

---

## 4. Component tests

- Storybook stories for every component; interaction tests (Storybook Test Runner).
- Axe a11y check per story (zero violations gate).
- Visual regression via Playwright + `pixelmatch`. Diffs above threshold fail CI.
- Reduced-motion + RTL + dark-mode variants captured.

---

## 5. End-to-end (Playwright)

Critical journeys, each ≤ 90 s:
1. Onboarding solo → S1 happy path → first action accept (mock-mode).
2. Tribe creation, two browsers, alignment widget converges.
3. Resource upload → ingestion → S3 unlock.
4. Action completion → token award → ledger reflects.
5. Empathy trigger → recovery offered → recovery accepted.
6. Voice mode round trip (mocked STT/TTS).
7. Goal achieved → VC issued → download badge.
8. Account deletion → row counts match expected.
9. RTL Arabic onboarding to S1.
10. Mentor read access — sees aggregates, blocked from raw chat.

E2E runs against staging with mocks for LLM and STT/TTS to be deterministic. Real-LLM smoke runs nightly only.

### 5.1 Multi-user
Playwright **fixtures** spin up two browser contexts to validate Realtime presence and alignment.

### 5.2 Mobile viewport
Each critical journey also runs at iPhone 13 viewport with touch emulation.

---

## 6. Performance tests

- **Lighthouse CI** thresholds: a11y 95, performance 80, best-practices 95, SEO 90.
- **k6** load tests on edge functions: 200 RPS sustained at p95 < 500 ms.
- **Cypress-bundle-analyzer**-equivalent check on web bundle budget.

---

## 7. Chaos & resilience

- LLM provider outage simulation: kill provider; system falls back to secondary; failing that, falls back to mocks; UI stays responsive.
- DB read replica lag simulation (where applicable).
- WebRTC drop / reconnect.
- Rate-limit storm (auto-throttle and circuit-break).

A monthly **GameDay** runs a scripted incident; engineers respond from runbooks; gaps are filed.

## 9. Test data

- Anonymised seed corpus per locale, generated by an offline script.
- "Canonical tribes" for replay: Amina (solo + companion), Mateo (solo + nudge), Lisbon trio (3 members), Wellness club (Dr Ngozi mentor).
- Test users have a stable seed so trace IDs reproduce across runs.

---

## 10. Local dev parity

- A `make test` runs the entire pyramid except E2E in < 5 min on a laptop.
- E2E runs against `docker compose up` of Supabase (auth, db, realtime) + edge.
- Real LLM disabled by default locally; opt-in via `USE_MOCKS=false`.

---

## 11. Quality gates per PR

A PR reaches `mergeable` only when:
- all CI checks green,
- coverage delta ≥ 0,
- LLM eval delta non-regressive on tier-0 scenarios,
- a11y unchanged or improved,
- bundle size delta ≤ +5 kB or explicitly justified,
- threat model not contradicted (privacy review label if data flows touched),
- one approval from a code owner.
